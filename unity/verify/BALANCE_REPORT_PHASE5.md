## §6.5 Pass/Fail 基準の判定

### a. 盗賊の各AI構成の中央値が上昇すること (必須) — **FAIL**

### d. 100階クリア率が警告ライン (10%) を超えないこと — **PASS**
  - 全15構成で勝率 10% 以下。

### 判定サマリ (5b時点): a FAIL / b PASS / c FAIL / d FAIL — §2.2 の初期重み表のままでは基準を満たさない

**1. 重み付けレバーの構造的限界 (最重要の発見)**
「毎階層クリアでドラフト」× プール18種のもとでは、長命キャラは1ランでプールを
ほぼ全取得する (魔法使い 17.0/18種、ナイト 19.2回=灯火スタック含む)。重み付けは
**取得の順序**しか変えられず**取得の集合**を変えられないため、§1 原則3 が想定した
「誰に何を見せるかで格差を縮める」レバーは、長命キャラに対しては実質無効化される。
c (ギャップ縮小) の FAIL はこの構造による必然で、重み表の数値チューニングでは解決しない。

**2. a FAIL (盗賊×Hard 中央値 2→2) の根本原因**
盗賊×Hard は中央値2階 = ドラフト機会が平均1.7回しかなく、「レリックが蓄積する前に死ぬ」。
ThiefRescue ×3.0 の優遇はドラフトが発生すれば効いている (Easy 7→12 / Normal 7→13 と
+71〜86% 上昇、a の 2/3 は上昇) が、Hard の序盤即死には間に合わない。
対策候補: §3.2 スターターパーク (ラン開始時装備) が本命。重み側の追加チューニングでは救えない。

**3. c/d FAIL (魔法使い独走・勝率>10%) の根本原因**
汎用強化 (#4 老練の構え=必要ターン-1 / #13 予備の呼吸=透明化+1 / #5 不死鳥 / #12 二段構え /
#10 護りの起点) が魔法使いの構造優位 (透明化7回/10階リセット) をそのまま増幅する。
General ×0.5 は取得を遅らせるだけで、86階も生きるランでは全て揃う。
勝率 10.7%/13.7% は §8 の「100階クリアはほぼ不可能」前提を崩すため d の警告ラインを超過。

**4. 次スプリント (ゲート判定) への提案 — いずれも設計変更を伴うためオーナー/Fable裁定マター**
- 案A: `RelicConfig.DraftInterval` の導入 (§2.1 が予告済みの定数化)。「6階層目以降・2階層に1回」等で
  取得総数を絞れば、重み=取得順序レバーが再び効くようになる (c/d 両方に効く)。
- 案B: 1ラン取得数の上限 (例: 8個)。ビルド選択の意思決定も生む。
- 案C: 魔法使い所持時、TurnCountReduction/SkillMaxUsageBonus 系 (#4/#13) を HardAICounter と
  同様の「重み0=除外」へ拡張 (§2.2 の明示ルール追加)。ただし1の構造限界により単独では不十分。
- 盗賊×Hard は案A〜Cでは救えない → スターターパーク (5c) とセットで再評価。

**5. 測定できていないもの (この表より実効果は上振れしうる)**
Tier2 3種 (#6/#15/#16) と #11 影の抜け道はシミュレータの前提上、効果ゼロとして扱われている。
それでもドラフト枠は消費している (=シム上は「ハズレ枠」) ため、実機ではこの表より
生存が伸びる方向のバイアスがかかる点に留意。

### 選定理由

**選定: `DraftInterval = 1`, `MaxRelicsPerRun = 4`**。優先度 c > d > a' > b で評価し、
同点の場合は最上位基準 (c) での僅差最小化を tiebreak とする。

1. **d はレバーで確実に解決できる。** (1,999) では2構成が勝率10%を超え「100階クリアはほぼ不可能」
   という §8 の前提を崩していたが、cap<=8 の全組 ((1,8)/(2,8)/(2,6)/(1,4)) で15構成すべて
   勝率10%以下になった。これは本タスクの導入目的 (長命キャラの際限ない snowball を止める) に
   そのまま合致する。
2. **d=PASS の4組 ((1,8)/(2,8)/(2,6)/(1,4)) は a/b/c/d のPASS/FAILパターンとしては同一
   (c-FAIL/d-PASS/a-FAIL/b-PASS) — 4組を区別するのは「c=FAILの中でどれだけギャップが小さいか」
   という同点判定 (tiebreak) であり、機械的に一意に決まるわけではない。** ただし優先度
   c > d > a' > b の c が最上位である以上、「c で最も僅差 (=改善余地が最大) な組を選ぶ」という
   tiebreak は優先度の定義と整合する選び方である。ギャップは (1,8)=52.8、(2,8)=52.7、
   (2,6)=47.6、(1,4)=39.7。**cap を絞るほどギャップは縮小し、同じ cap では interval を2にしても
   ほぼ効果がない** ((1,8) の52.8 と (2,8) の52.7 はほぼ同値)。つまり実際にギャップを縮めている
   のは cap (MaxRelicsPerRun、案B) であり、interval (DraftInterval、案A) の寄与は本シミュレーション
   の範囲では検出できないほど小さい。そのため本tiebreakでは cap を最も絞れている (1,4) を選ぶ。
   ただし「richer builds」を優先するなら (1,8) が正当な代替案である (§[要検証] 参照)。
3. **c は本レバー単体では構造的に到達不能に近く、(1,4) でも FAIL のまま。** c の判定式は
   「レリック有効時のギャップ」を「レリック無効 (baseline) 時のギャップ 38.8」と比較する。
   しかし baseline 時点で既に魔法使いは他キャラ平均より38.8階層分優位 (透明化7回/10階リセット
   という**ベースゲームの強さ**であり、レリックとは無関係)。レリックという「追加の報酬系」
   だけでこの土台の格差を割ることは原理的に難しく、cap を 8→6→4 と絞ってもギャップは 38.8
   近辺で頭打ちになる ((1,4) で 39.7)。cap=2 まで絞るとようやく数値上 38.7 で PASS になるが、
   これは「レリックをほぼ無効化して baseline に近づける」という退化 (盗賊中央値も8〜9まで
   落ち込み、取得レリックは約2個/ランのみ、勇者は10→14止まり = Phase 5 が目指した
   「ローグライク深化」が実質消える) であり、0.1差の非単調な閾値越え (cap=1では39.5に戻り、
   cap=3では39.9に戻る) は統計的に頑健な解ではなく数値の偶然の産物と判断し、採用しなかった。
   **c=FAIL は本レバーの限界として受け入れ、5bの案C (魔法使い所持時の汎用強化タグをさらに絞る)
   や、より根本的にはベースゲーム側の魔法使い弱体化を、次スプリントの [要検証] 事項として送る。**
4. **a' はどの d=PASS 組でも同じ (盗賊 10/10) であり、(1,4) を選ぶ上での障害にならない。**
   盗賊の中央値は baseline 7 → (1,999) 12/13 → d=PASS 組はすべて 10/10。(1,999) のピーク値
   (12/13) は失うが、baseline からの改善 (+3) はどの d=PASS 組でも同程度に維持している。
5. **b は全組でPASS**のため選定に影響しない。
6. **DraftInterval (案A) の結論**: 本タスクは2レバーを実装したが、実測データが支持する最終値は
   `DraftInterval=1` (=毎階層、5b当初の挙動のまま) だった。「頻度を絞る」効果は「1ラン取得数を
   絞る」効果に比べて弱く、両方を同時に絞ると (2,6) のように cap 側の絞り込みが甘くなり
   (同じ「ドラフト提示のペース」を維持しようとすると cap を上げがちになる)、結果的に c が
   悪化する。`RelicConfig.DraftInterval` はインフラとして残し、実機プレイテストで
   「毎階層はドラフトUIが煩雑」と判明した場合に備える。

### [要検証] まとめ

- **c (魔法使いギャップ縮小) は本レバーのみでは達成できない。** 「レリック無効時点の格差」を
  基準にする現行の c の定義そのものを見直すか (例: (1,999) の 56.7 のような「レリックによる
  暴走」を基準にした相対評価にする)、5bの案C (魔法使い所持時の General タグを HardAICounter
  同様に重み0へ拡張) や、レリックとは独立した魔法使いのベースキット弱体化を検討する必要がある。
  オーケストレータ/オーナー裁定マター。
- **盗賊Hardの底上げ (a) は本レバーのスコープ外のまま。** 中央値2階 = ドラフト機会がほぼ無い
  ことが根本原因であり、5bの結論どおり §3.2 スターターパーク (ラン開始時装備) が本命。
- **1ラン平均4個 (relics/run 3.2〜4.0) は「深化」としてはやや薄いという設計上のトレードオフ。**
  d を確実に解決する範囲で c (ギャップ) を最小化した tiebreak の結果であり、実機プレイテストで
  「レリックが少なすぎてビルドの手応えがない」という声が出た場合、**正当な代替案は (1,8)**
  (盗賊Normal 11 で (1,4) の10よりわずかに良く、取得レリックは5.7〜8.0/ランと大幅に豊か。
  ただし魔法使いギャップは52.8とc改善幅は(1,4)より大きく後退する) への切り替えである。
  d の許容ライン (勝率10%) 自体を緩める案も含め、再裁定が必要になる可能性がある。
